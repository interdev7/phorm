# Анализ поддержки диалектов и валидаторов в PHORM (Актуализировано)

В ходе работы над проектом был проведен детальный анализ кодовой базы проектов `phorm`, `phorm_sqlite`, `phorm_annotations`, `phorm_generator` на предмет совместимости с SQL-диалектами и корректности генерации ограничений базы данных. Ниже представлены результаты анализа, статус исправления выявленных проблем и текущее состояние архитектуры.

---

## 1. Проблема с плейсхолдерами (Positional Placeholders)

### Что заявляла документация:
В `docs/01-overview.md` (раздел *Dialects & Pluggable SQL Architecture*) утверждалось:
> *Positional Placeholders: SQLite uses `?`, Postgres uses `$1`, `$2`, etc.*
> *The query builder, JSON eager loading system, and column compiler in phorm never generate hardcoded database-specific SQL strings. Instead, they delegate to the active SqlDialect...*

### Выявленная проблема:
- Ранее метод `compilePlaceholder(int index)` из интерфейса `SqlDialect` ни разу не вызывался в ядре ORM.
- Класс `WhereBuilder` в `lib/src/where_builder.dart` жестко кодировал плейсхолдер `?` во всех методах фильтрации, а замена в условиях приводила к бесконечному циклу в случае SQLite (так как `replaceFirst('?', '?')` заменял знак вопроса на самого себя).
- Из-за этого библиотека не могла работать с СУБД вроде PostgreSQL (где требуются плейсхолдеры `$1`, `$2`).

### Статус: ИСПРАВЛЕНО ✅
- В `WhereBuilder._Condition.compile` внедрена динамическая компиляция плейсхолдеров с использованием `dialect.compilePlaceholder(...)`.
- Реализована безопасная обработка плейсхолдеров, равных `'?'` (например, для SQLite и MySQL): в этом случае вместо бесконечного цикла замены строки мы просто подсчитываем количество `'?'` в SQL-условии и безопасно инкрементируем `paramIndex.value` на это количество. Это исключило любые зависания тестов и обеспечило корректную работу.
- Для PostgreSQL и других СУБД с позиционными плейсхолдерами (например, `$1`, `$2`) цикл замены продолжает работать в штатном режиме.

## 2. Экранирование идентификаторов (Identifier Escaping)

### Что заявляла документация:
В `docs/01-overview.md` утверждалось:
> *Identifier Escaping: SQLite/Postgres uses `"table"."column"`, while MySQL uses `` `table`.`column` ``.*

### Выявленная проблема:
- Метод `escapeIdentifier(String name)` из интерфейса `SqlDialect` не использовался.
- В ядре `lib/src/core.dart` при сборке JOIN-запросов и JSON-агрегации имена колонок и таблиц объединялись обычным соединением строк через точку без экранирования, что приводило к синтаксическим ошибкам при совпадении с зарезервированными ключевыми словами (например, `user`, `order`).

### Статус: ИСПРАВЛЕНО ✅
- Сборщик JOIN-запросов и JSON-агрегации в ядре ORM (`lib/src/core.dart`) переведен на обязательное экранирование идентификаторов через `activeDialect.escapeIdentifier(columnName)`.
- Все имена таблиц и столбцов в генерируемых SQL-запросах теперь надежно оборачиваются в кавычки согласно требованиям используемой СУБД (например, `"orders"."total"` для SQLite).

## 3. Специфика генератора схем и триггеров

### Что заявляет документация:
PHORM позиционируется как независимая от конкретной СУБД (driver-agnostic) библиотека.

### Реальное положение дел (Ограничение кодогенератора):
- Генератор DDL-схем (`phorm_generator`) на данный момент называется `SqliteSchemaGenerator` и ориентирован на генерацию SQLite-совместимых схем:
  - Типы полей по умолчанию маппятся в SQLite-типы (например, `TEXT` для DateTime).
  - Генерация триггеров обновления дат (`updated_at`) жестко прописана под синтаксис SQLite с использованием встроенной функции `datetime('now')`.
- **Следствие**: Хотя сам рантайм-сборщик запросов (Query Builder) теперь полностью абстрагирован от СУБД и готов к поддержке любых диалектов (PostgreSQL, MySQL и др.), кодогенератор схем (`phorm_generator`) генерирует схему DDL только под SQLite. Для использования других СУБД (например, PostgreSQL) в будущем потребуется альтернативный генератор DDL-схем либо ручное описание таблиц и триггеров через файлы миграций.

### Статус: ЗАДОКУМЕНТИРОВАНО ✅
- Мы провели тщательный аудит и отразили это архитектурное ограничение в официальной документации:
  - Добавили раздел `Schema Generator is SQLite-Specific` в [09-pitfalls-and-limitations.md](file:///Users/interdev7/Documents/phorm/docs/09-pitfalls-and-limitations.md).
  - Добавили примечание в дорожную карту диалектов в [01-overview.md](file:///Users/interdev7/Documents/phorm/docs/01-overview.md), честно предупреждая разработчиков о специфике генерируемого SQL DDL.

## 4. Проблемы с валидатором `NotEmptyValidator`

### Выявленная проблема:
- Ранее в некоторых местах использовался жестко заданный синтаксис пустой строки с двойными кавычками `col <> ""`, что нарушает стандарты большинства СУБД (где двойные кавычки экранируют идентификаторы, а строковые литералы должны быть в одинарных кавычках `''`).
- Существовал риск, что кастомный валидатор `NotEmptyValidator` не будет корректно распознан генератором как ISqlValidator для CHECK-ограничений в базе данных, если он объявлен с использованием вычисляемого геттера `get sql` вместо константного `final` поля.

### Статус: ИСПРАВЛЕНО ✅
- Класс `NotEmptyValidator` приведен к стандарту:
  `const NotEmptyValidator({this.constraint}) : sql = "{column} <> ''";`
- Поле `sql` теперь является константным полем `final String sql`, что позволяет кодогенератору успешно считывать и встраивать CHECK-ограничение (`CHECK(col <> '')`) в итоговую схему базы данных.
- Документация по валидаторам ([10-validators.md](file:///Users/interdev7/Documents/phorm/docs/10-validators.md)) обновлена и ссылается на корректный синтаксис с одинарными кавычками.

## Резюме

В результате проделанной работы:
1. Архитектура диалектов PHORM была полностью интегрирована в сборщик запросов (Query Builder) и `WhereBuilder`.
2. Избавились от жестко зашитых плейсхолдеров `?` в рантайме, сделав сборку запросов полностью pluggable.
3. Документация была детально скорректирована, чтобы честно отражать специфику работы генератора схем и помочь разработчикам избежать непредвиденного поведения при проектировании кроссплатформенных приложений.
